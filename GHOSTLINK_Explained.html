How Messages Work — From Start To Finish

1 You Register

You open GHOSTLINK and type a made-up username and a password. On your computer only:

• A 384-bit secret key is created — a number so big that guessing it would take billions of years with every computer on Earth.

• A matching public key is created — like a padlock you can give to anyone. Only your secret key opens it.

• Your password is scrambled 600,000 times before being sent. The server stores only the scrambled result.

• Your computer sends: username + scrambled password + device name + public padlock.

The server now knows: a made-up username, scrambled math, and a public padlock. None of this can be used to impersonate you or read your messages.

2 You Find A Friend

You type a username in the search box. The server says "yes, that user exists" and returns their username.

You click their name. Your computer asks the server for their public padlock(s).

Now your computer has their padlock. The server never knows who you're searching for beyond what you type — and it doesn't log searches.

3 You Send A Message

You type "hey what's up" and click Send. Everything below happens on your computer, before anything leaves:

• Your computer does ECDH math: your secret key + their public padlock = a shared secret that only you and the recipient can compute. The server never sees this.

• Your computer builds a package: the message body, your display name, any note/comment, your device ID, and a timestamp.

• Your computer encrypts the entire package with AES-256-GCM using the shared secret. The result is complete gibberish.

• This gibberish is wrapped in an envelope — just an ID, a timestamp, a random number, and the scrambled data.

• The envelope is sent to the server.

4 Your Friend Gets The Message

Every 4 seconds, your friend's computer checks the server: "Any new messages for me?"

• The server finds your envelope and hands it over. Then the server DELETES it permanently — gone from the database forever.

• Your friend's computer does the ECDH math in reverse: your public padlock + their secret key = the same shared secret.

• Their computer decrypts the gibberish and displays: your display name, your message, and any note you attached.

The message now exists ONLY on your two devices. The server has zero copies. Zero record. Nothing.

How File Sharing Works

1 You Attach a File

You click Attach and pick any file — a photo, video, document, anything. No size limit.

• Your computer reads the entire file.

• Your computer encrypts it with AES-256-GCM using the shared secret. The file becomes random bytes.

• Your computer uploads the encrypted blob to the server. The server stores it under a random name like a3f8b2c1.enc.

The server never learns: the original filename, the file type, the file size (in plaintext), or what's inside. It sees only random bytes.

• Your computer sends a message to your friend with the file ID and the original filename — encrypted inside the message, so the server never sees the filename either.

2 Your Friend Downloads

• Their computer receives the file notification message and decrypts it — now it knows the file ID and the original filename.

• Their computer downloads the encrypted blob from the server.

• The moment the download completes, the server deletes the file from disk and removes it from the database.

• Their computer decrypts the blob with the shared secret and saves it with the original filename.

• One download only. If your friend tries again, the server returns "410 Gone" — the file no longer exists.

• If your friend doesn't download within 24 hours, the server automatically deletes the expired file.

What Happens If...

The server gets hacked?
Hackers get encrypted garbage and random filenames. Useless without client keys — which were never on the server.
Someone steals your computer?
Your keys are sealed by the TPM 2.0 chip and DPAPI — they can't be extracted. You can remotely NUKE your data by clicking one button in Settings.
Law enforcement demands server data?
Messages are already deleted. Files are already deleted. The server has no message history, no plaintext, no filenames, no real identities. There is nothing useful to hand over.
Someone intercepts network traffic?
Everything is AES-256-GCM encrypted before leaving your computer. They see only encrypted data streams.
A quantum computer is built?
AES-256-GCM (message encryption) remains safe — 2^128 effective strength. Only the ECDH key exchange would need upgrading to hybrid post-quantum (ML-KEM-1024), which is a straightforward update.
The server goes down?
Your client detects it immediately, blocks all messages, and shows "ENCRYPTION SERVER OFFLINE — Messages blocked for security." It retries every 3 seconds until the server is back. No unencrypted message ever leaves your computer.
